# AREX - System Architecture & Monorepo Philosophy

This document outlines the high-level architecture of AREX, focusing on the monorepo organization, core structural boundaries, and safety-critical compliance design.

## Monorepo Philosophy & Organization

AREX is structured as a unified monorepo to solve a critical regulatory challenge: **preventing API drift and ensuring perfect compliance traceability**. By housing all application components under a single repository, we gain several key benefits:

1. **Single Source of Truth**: The `shared/openapi/` folder holds the API contract (`arex.yaml`). Both the Python backend and TypeScript frontend are built, verified, and typed against this file.
2. **Cohesive Infrastructure**: Dockers and database configuration live in `infra/` so that a single `docker-compose.yml` orchestrates the entire platform locally and in staging environments.
3. **Day-One Regulatory Tracing**: `docs/compliance/21-cfr-part-11-mapping.md` provides audit-ready trace logs mapping code capabilities directly to federal regulations.

## Core Architectural Boundaries

AREX utilizes a decoupled architecture to maintain high availability, security, and strict data validation:

```
                  ┌──────────────────────┐
                  │   Next.js Frontend   │
                  └──────────┬───────────┘
                             │
                      JSON / HTTP (REST)
                             │
                             ▼
                  ┌──────────────────────┐
                  │   FastAPI Backend    │
                  └────┬────────────┬────┘
                       │            │
             Read/Write│            │Orchestration
                       ▼            ▼
         ┌────────────────┐      ┌────────────────┐
         │ PostgreSQL DB  │      │ LangGraph AI   │
         │ (Deterministic)│      │  (Agentic)     │
         └────────────────┘      └───────┬────────┘
                                         │
                                         ▼
                                 ┌───────────────┐
                                 │   Qdrant DB   │
                                 │ (Vector Store)│
                                 └───────────────┘
```

### 1. Deterministic vs. Agentic Separation
- **Traditional Backend (`backend/app/services/`)**: Implements strict deterministic workflows, e.g., role-based access control (RBAC), database transaction processing, and rule-based compliance scoring.
- **AI/Agentic Layer (`backend/app/ai/`)**: Implements complex reasoning using LangGraph and the Qwen3 8B LLM. This layer has zero direct access to modify canonical system states or documents without human intervention.

### 2. Safety-Critical Human-in-the-Loop Workflow
Under FDA 21 CFR Part 11 guidelines, AI cannot write or edit standard operating procedures (SOPs) or sign off on updates autonomously.
- The `ApprovalWorkflow` state machine enforces that all remediation drafts generated by the `RemediationAgent` remain in a `PENDING_REVIEW` state.
- A human administrator must explicitly verify, edit, and sign off (with digital signature) on the changes.
- Only approved drafts can be exported via the `ExportService` or committed to the active Knowledge Base.

## Database & Isolation Strategy
- **Postgres**: Serves as the primary transaction database, storing organizations, users, document metadata, audit logs, and approval records. Row-Level Security (RLS) or organizational scoping prevents cross-tenant data leaks.
- **Qdrant**: Stores vector embeddings (generated by the BGE-M3 model) of chunks from ingested company SOPs. This enables RAG retrieval for context-grounded agent suggestions.
